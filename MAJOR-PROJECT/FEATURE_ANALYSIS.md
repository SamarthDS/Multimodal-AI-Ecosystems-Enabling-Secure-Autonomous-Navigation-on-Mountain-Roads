## 3. Key Technical Takeaways

### Why the implemented features work

- **Simplicity-first architecture**: A single TCP socket on `localhost` provides reliable, low-latency bidirectional communication without the complexity of broadcast protocols.
- **BeamNG AI leveraged, not fought**: The system works _with_ BeamNG's AI modes (`chase`, `stop`, `random`) and lane-enforcement commands rather than attempting low-level vehicle control.
- **Kinetic-energy negotiation is deterministic**: Using speed comparison + VID tie-breaker ensures exactly one car yields in every scenario, eliminating deadlocks.
- **Metrics are first-class citizens**: Every event is timestamped with `time.perf_counter()`, and CSV logging is automatic — enabling statistically rigorous, multi-run experimental analysis.

### Why the unimplemented features could not be developed

- **Engine limitations**: BeamNG's Torque3D engine imposes strict terrain format requirements, single-tile collision constraints, and lacks programmable infrastructure — blocking realistic map import, multi-lane scenarios, and V2I.
- **Hardware dependency**: DSRC/C-V2X, OBU/RSU hardware, and PKI security stacks require physical equipment and standardized protocol implementations that cannot exist in a software-only simulation.
- **Physics fidelity trade-offs**: BeamNG.tech's sensor suite was available but introduced physics-timestep mismatches and terrain glitches that invalidated baseline measurements — making it an unreliable upgrade path.
- **API maturity**: At the time of development, `beamngpy` lacked APIs for traffic fleet management, weather parameterization, and non-vehicle actors (pedestrians/cyclists).
- **Scope and timeline**: Features like LiDAR pipelines, ASN.1 message encoding, and network-channel modeling are each multi-month efforts that exceeded the project's academic timeline and budget.
